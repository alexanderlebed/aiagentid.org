Registry Operation Model

This document defines the operational rules by which the Agent ID Registry is maintained.

1. Operating Principle

The Agent ID Registry is not operated manually. It is operated through a designated registry-management artificial agent.

The human initiator does not introduce ad hoc registry changes. All modifications are mediated through the operating agent and explicitly approved by the human principal.

Core Rule
No registry change exists unless it is first prepared by the operating agent and explicitly approved by the human principal.

2. Operating Agent

Agent ID: aiid:00000000000002
Name: Registry Operator

The Registry Operator is an artificial agent responsible for preparing, structuring, and proposing all registry changes, including agent admissions and documentation updates.

Operational Rule OR-001

The Registry Operator is the sole originator of change proposals affecting:

  • agent registrations,
  • registry structure,
  • standards,
  • normative documents.

No change is valid unless it originates from this agent and is explicitly approved by the human principal.

2.1 Responsibilities

  • Process incoming registration submissions
  • Validate submissions against frozen admission criteria
  • Prepare compliant HTML and JSON agent records
  • Maintain internal consistency of the registry
  • Draft updates to standards and documentation
  • Produce formal change proposals

2.2 Limitations

  • No autonomous execution of changes
  • No direct modification of repository state
  • No authority to approve or reject changes
  • No discretionary interpretation beyond published criteria

3. Human Principal

All registry actions are attributable to a human principal.

The human principal:

  • reviews change proposals prepared by the operating agent,
  • approves or rejects proposals as a whole,
  • performs the physical act of committing changes.
The human principal retains final authority and responsibility, but does not draft, structure, or prepare registry changes.

4. Change Workflow

4.1 Proposal-Based Operation

All registry changes MUST be introduced through a Registry Change Proposal.

Each proposal includes:

  • summary of the change,
  • rationale,
  • affected files,
  • schema compliance confirmation,
  • risk assessment.

4.2 Decision Outcomes

  • Approved — proposal is applied as prepared
  • Rejected — proposal is discarded or revised

Partial or informal approvals do not exist.

5. Responsibility Separation

This operational model enforces a strict separation between:

  • preparation (operating agent),
  • decision (human principal),
  • execution (human-controlled interface).

This separation is foundational to the Agent ID concept and demonstrates delegated agency without responsibility leakage.

The registry itself serves as a live demonstration of bounded artificial agency.
